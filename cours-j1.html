📝 Prérequis

Vous venez de terminer la POO en Java : vous savez ce qu'est une classe, un objet, un attribut. C'est exactement le pont qu'on va emprunter aujourd'hui. Un peu de logique et d'algorithmique suffisent pour le reste. Aucune connaissance BDD n'est attendue.

Fil rouge de tout le module : BiblioTech, la modernisation d'une bibliothèque universitaire qui gère 50 000 livres… avec des fiches cartonnées depuis 1995. On va remplacer ces tiroirs par une vraie base de données.

Pourquoi une base de données ?

Le problème : les fiches cartonnées

🪄 La métaphore

Imaginez la bibliothèque de BiblioTech : trois tiroirs remplis de fiches cartonnées. Pour savoir si « Clean Code » est disponible, Sophie fouille pendant 15 minutes. La fiche dit « disponible »… mais le livre a été emprunté sans que personne ne le note. Résultat : 200 livres perdus dans l'année.

Une base de données, c'est ce même tiroir de fiches — mais rangé, indexé, jamais en double, et interrogeable en une fraction de seconde.

Définition. Une base de données (BDD) est un ensemble de données organisées pour être stockées durablement et retrouvées rapidement et de façon fiable. Sans elle, impossible de tenir un e-commerce, une banque ou même une petite appli mobile.

Concrètement, une BDD sert à :

  • Stocker des millions d'informations sans les perdre ;
  • Organiser ces informations pour éviter les doublons et les erreurs ;
  • Retrouver n'importe quelle donnée instantanément.
🎯 Pourquoi on fait ça

100 % des applications professionnelles utilisent au moins une base de données. Un développeur backend passe environ 70 % de son temps à interagir avec la BDD. Un e-commerce moyen exécute plus de 10 000 requêtes par minute. Bref : savoir modéliser des données, c'est le socle du métier.

Le SGBD : le magasinier

🪄 La métaphore

Les fiches ne se rangent pas toutes seules. Il faut un magasinier qui reçoit les demandes, va chercher la bonne fiche, la modifie, et empêche deux personnes de la modifier en même temps. Ce magasinier, c'est le SGBD.

Définition. Un SGBD (Système de Gestion de Base de Données) est le logiciel qui gère la base à votre place. Exemples : MySQL, MariaDB, PostgreSQL, Oracle, SQL Server. Il assure quatre missions :

  • Stocker les données de façon structurée et persistante ;
  • Interroger les données via un langage dédié, le SQL (« Structured Query Language », c'est-à-dire « langage structuré de requêtes ») ;
  • Modifier les données (ajout, mise à jour, suppression) ;
  • Sécuriser & garantir l'intégrité (droits d'accès, cohérence).
💡 Bon à savoir

Le mot « base de données » désigne les données. Le SGBD désigne le logiciel qui les gère. On dit « la base bibliotech tourne sous MySQL » : bibliotech est la base, MySQL est le SGBD.

Relationnel (SQL) ou NoSQL ?

Il existe deux grandes familles de bases. Vous travaillerez sur la première, mais il faut savoir situer la seconde.

FamillePrincipeExemplesIdéal pour…
Relationnel (SQL)Données rangées en tables liées entre ellesMySQL, PostgreSQL, OracleDonnées structurées avec relations claires (banque, e-commerce, bibliothèque)
NoSQLDocuments, clé-valeur, colonnes ou graphesMongoDB, Redis, Neo4jStructure très souple, volumes gigantesques, cache
⚠️ La confusion classique

NoSQL ne veut pas dire « sans base de données » ni « plus moderne donc mieux ». Ça signifie « Not Only SQL ». BiblioTech est relationnel : un emprunt DOIT être relié à UN livre et UN adhérent — c'est précisément ce que le modèle relationnel garantit.

Le vocabulaire des bases relationnelles

Cinq mots à maîtriser. On les file avec la métaphore du tiroir de fiches.

Table, ligne, colonne

🪄 La métaphore

Une table = un tiroir ne contenant que des fiches du même modèle (toutes les fiches « livre »). Une ligne = une fiche précise (le livre « Clean Code »). Une colonne = une rubrique imprimée sur toutes les fiches (le titre, l'auteur, l'année…).

TermeAussi appeléDéfinitionExemple BiblioTech
TableRelationStructure qui stocke des données de même naturela table livres
LigneEnregistrementUne entrée précise dans la tablele livre « Clean Code »
ColonneChampUne propriété commune à toutes les lignestitre, auteur, annee

Exemple concret — la table livres :

id_livre 🔑titreauteurisbnannee
1Clean CodeRobert C. Martin978-01323508842008
2Design PatternsGang of Four978-02016336101994
3The Pragmatic ProgrammerHunt & Thomas978-01359570592019

Chaque ligne est un livre. Chaque colonne est une rubrique. La colonne id_livre porte la clé 🔑 : on l'explique juste après.

La clé primaire (PK)

🪄 La métaphore

Deux livres peuvent avoir le même titre, le même auteur, la même année. Pour ne jamais les confondre, on leur donne un numéro unique, comme un numéro de carte d'identité. Ce numéro, c'est la clé primaire.

Définition. La clé primaire (PK, « Primary Key », c'est-à-dire « clé primaire ») est la colonne qui identifie chaque ligne de façon unique. Ici : id_livre.

⚠️ Le piège

Deux lignes ne peuvent jamais partager la même clé primaire, et elle ne peut jamais être vide. C'est la règle qui garantit qu'on parle bien du bon livre. Si le numéro pouvait se répéter, on retomberait dans le chaos des fiches cartonnées.

La clé étrangère (FK)

🪄 La métaphore

Sur une fiche « emprunt », au lieu de recopier tout le dossier de l'adhérent, on écrit juste : « voir l'adhérent n° 42 ». Ce renvoi vers une autre fiche, c'est la clé étrangère.

Définition. La clé étrangère (FK, « Foreign Key », c'est-à-dire « clé étrangère ») est une colonne qui pointe vers la clé primaire d'une autre table. Exemple : dans la table emprunts, la colonne id_adherent renvoie vers la table adherents.

💡 L'idée clé

La clé étrangère, c'est le lien entre deux tables. C'est ce qui rend une base « relationnelle » : les tables ne vivent pas isolées, elles se référencent entre elles au lieu de recopier les mêmes informations partout.

Le pont avec la POO Java

🪄 La métaphore

C'est le même objet vu dans deux langues. En Java, vous décrivez une class Livre puis vous fabriquez des livres avec new. En base de données, vous décrivez une table livres puis vous ajoutez des lignes. Même idée, autre vocabulaire.

📝 Rappel POO (vu juste avant)

Une classe est un moule qui décrit des attributs. Un objet est un exemplaire concret créé à partir du moule. Gardez ces deux mots en tête : ils ont un jumeau côté base de données.

La correspondance à retenir :

En POO JavaEn base de donnéesRôle commun
Une classe LivreUne table livresLe moule : décrit la structure
Un objet (new Livre(...))Une ligne (enregistrement)Un exemplaire concret
Un attribut (String titre;)Une colonne (titre)Une propriété
Le type Java (int, String…)Le type SQL (INT, VARCHAR…)La forme de la donnée
Créer : new Livre(...)Ajouter : INSERT INTO livres …Fabriquer un exemplaire

Exemple côte à côte :

// En Java : une classe décrit la structure
class Livre {
    int id;
    String titre;
    int annee;
}
// On fabrique un exemplaire
Livre l = new Livre(1, "Clean Code", 2008);
-- En SQL : une table décrit la même structure
CREATE TABLE livres (
    id_livre INT,
    titre    VARCHAR(255),
    annee    INT
);
-- On ajoute un exemplaire (= une ligne)
INSERT INTO livres VALUES (1, 'Clean Code', 2008);

Analysons ces deux blocs. À gauche, class Livre déclare trois attributs typés ; à droite, CREATE TABLE livres déclare trois colonnes typées : c'est le même moule. En bas, new Livre(...) et INSERT INTO ... VALUES(...) font la même chose : créer un exemplaire concret avec des valeurs réelles.

⚠️ Là où l'analogie s'arrête

Une table n'a pas de méthodes (pas de comportement, seulement des données). Et surtout : en Java, un objet Emprunt pointerait directement vers un objet Adherent en mémoire. En base, ce lien passe par une clé étrangère (un simple numéro dans une colonne), pas par une référence d'objet. On garde l'analogie pour comprendre, pas pour tout calquer.

La méthode Merise : penser avant de coder

Merise, c'est quoi ?

🪄 La métaphore

On ne construit pas une maison en posant des briques au hasard. On commence par un plan d'architecte : où vont les murs, les portes, les pièces. Merise, c'est le plan d'architecte de votre base de données. On dessine avant de bâtir.

Définition. Merise est une méthode française de conception de bases de données, née dans les années 1970. Son principe : séparer clairement les données (ce qu'on stocke) des traitements (ce qu'on en fait), et modéliser du plus abstrait au plus concret.

💡 Pourquoi dessiner d'abord

Dans les ESN (Capgemini, Sopra Steria…), le plan est exigé et validé par le client avant tout code. Il vit ensuite 10 à 20 ans. Une banque a même migré d'Oracle vers PostgreSQL en régénérant sa base à partir de ce plan — 6 mois de travail économisés. Le plan survit à la technologie.

Les trois niveaux : MCD → MLD → MPD

Merise descend en trois marches, du « quoi » vers le « comment » :

NiveauModèleQuestionÀ quoi ça ressemble
ConceptuelMCD — Modèle Conceptuel de Données« QUOI stocker ? »Un schéma métier, sans aucune technique
LogiqueMLD — Modèle Logique de Données« COMMENT l'organiser en tables ? »La liste des tables, clés primaires et étrangères
PhysiqueMPD — Modèle Physique de Données« COMMENT l'implémenter ? »Le script CREATE TABLE pour MySQL
🎯 À retenir

MCD = le plan (indépendant de toute techno) → MLD = les tables (encore sur papier) → MPD = le SQL réel. Aujourd'hui on se concentre sur les deux premières marches : le MCD, puis sa traduction en MLD.

Le MCD en détail

Le MCD se dessine avec quatre briques : entités, attributs, identifiants, associations.

L'entité

🪄 La métaphore

Une entité, c'est le moule — exactement comme une classe Java. « CLIENT » est un moule : tous les clients auront un nom, un prénom, un email. On ne dessine pas chaque client, on dessine le moule.

Définition. Une entité est un objet du monde réel qu'on veut modéliser. On la dessine dans un rectangle et on la nomme en MAJUSCULES : CLIENT, PRODUIT, COMMANDE, LIVRE, ADHERENT.

L'attribut

🪄 La métaphore

Les attributs, ce sont les rubriques à remplir sur la fiche du moule. Pour CLIENT : nom, prénom, email, adresse.

Définition. Un attribut est une propriété qui caractérise une entité. On les liste à l'intérieur du rectangle. Ce sont les futures colonnes.

L'identifiant

🪄 La métaphore

Parmi les rubriques, une seule sert de numéro unique (comme la carte d'identité de la section 2). Dans un MCD, on la souligne.

Définition. L'identifiant est l'attribut souligné qui distingue chaque occurrence de façon unique : id_client, id_produit. Lors du passage au MLD, l'identifiant deviendra la clé primaire.

⚠️ Le piège de l'identifiant « naturel »

Tentant d'utiliser l'ISBN comme identifiant d'un LIVRE. Mais un ISBN peut être mal saisi, changer d'édition, ou manquer sur un vieux livre. On préfère souvent un id_livre artificiel (un numéro généré automatiquement) et on garde l'ISBN comme simple attribut. Règle générale : un identifiant doit être unique, stable et jamais vide.

L'association

🪄 La métaphore

Deux moules, ça ne suffit pas : il faut dire comment ils se parlent. Un CLIENT PASSE une COMMANDE. Ce verbe qui relie deux entités, c'est l'association. On la dessine dans un ovale et on la nomme… par un verbe.

Définition. Une association est un lien logique entre deux entités (ou plus), nommé par un verbe : PASSER, CONTENIR, EMPRUNTER, ECRIRE.

Exemple concret — une mini-boutique en ligne : un client passe des commandes, une commande contient des produits.

💡 L'attribut porté par l'association

Regardez quantite sur l'association CONTENIR. Pourquoi n'est-il ni dans COMMANDE ni dans PRODUIT ? Parce qu'il dépend des deux à la fois : c'est la quantité de ce produit dans cette commande. Un attribut qui ne peut vivre dans aucune des deux entités appartient à l'association qui les relie.

Les cardinalités : « combien de fois ? »

🪄 La métaphore

Un professeur suit plusieurs élèves, mais chaque élève n'a qu'un professeur principal. La cardinalité, c'est cette règle du « combien » qu'on note sur chaque bout de l'association, sous forme d'un couple (minimum, maximum).

Définition. Une cardinalité indique combien de fois au minimum et au maximum une occurrence d'entité participe à l'association. Les quatre couples usuels :

CardinalitéSe litExemple
(0,1)zéro ou un (optionnel, unique)Un employé a zéro ou un bureau attitré
(1,1)exactement un (obligatoire, unique)Une commande appartient à exactement un client
(0,n)zéro ou plusieurs (optionnel, multiple)Un client peut passer zéro ou plusieurs commandes
(1,n)un ou plusieurs (obligatoire, multiple)Une commande contient au moins un produit
🎯 Comment les lire

On lit toujours du point de vue de l'entité :

« Un CLIENT peut PASSER combien de COMMANDES ? » → de 0 à plusieurs → (0,n) côté CLIENT.
« Une COMMANDE est PASSÉE par combien de CLIENTS ? » → exactement 1 → (1,1) côté COMMANDE.

⚠️ L'erreur qui coûte cher

Une startup e-commerce avait modélisé « une commande a UN seul produit » — (1,1). Six mois plus tard, pour permettre plusieurs produits par commande, il a fallu refactoriser toute la base : 3 semaines de dev + migration des données. Prendre 5 minutes pour poser la bonne cardinalité au départ évite des semaines de galère ensuite.

💡 Le réflexe

Interrogez chaque côté séparément, à voix haute, avec un vrai exemple. Le minimum répond à « est-ce obligatoire ? » (0 = non, 1 = oui). Le maximum répond à « peut-il y en avoir plusieurs ? » (1 = non, n = oui).

Du MCD au MLD : vers les tables réelles

🪄 La métaphore

Le plan d'architecte (MCD) est validé. On passe au gros œuvre : on transforme les pièces dessinées en vrais murs. Le MLD, ce sont les tables concrètes avec leurs clés primaires et étrangères. Trois règles suffisent.

Règle 1 — Chaque entité devient une table

L'entité devient une table ; son identifiant souligné devient la clé primaire. CLIENT → table CLIENT (id_client, nom, prenom, email, adresse).

Règle 2 — Une association (1,1)↔(0,n) : la clé migre

Quand un côté est (1,1), la clé primaire de l'autre côté descend comme clé étrangère dans la table du côté (1,1). Ici COMMANDE (côté 1,1) hérite de id_client.

💡 Le sens de la migration

La clé va toujours vers le côté « exactement un ». Logique : chaque commande n'ayant qu'UN client, on peut ranger ce client directement sur la fiche commande. L'inverse serait impossible : un client a plusieurs commandes, on ne saurait pas laquelle inscrire.

Règle 3 — Une association (n,n) : on crée une table associative

Quand les deux côtés sont multiples (par ex. (1,n) ↔ (0,n)), l'association devient elle-même une table. Elle contient les clés des deux entités + les attributs de l'association. CONTENIR devient LIGNE_COMMANDE.

Le MLD complet de notre exemple :

-- # signale une clé étrangère (FK)
CLIENT (id_client, nom, prenom, email, adresse)
    PK : id_client

COMMANDE (id_commande, date_commande, statut, #id_client)
    PK : id_commande
    FK : id_client → CLIENT(id_client)

PRODUIT (id_produit, nom, description, prix)
    PK : id_produit

LIGNE_COMMANDE (#id_commande, #id_produit, quantite)
    PK : (id_commande, id_produit)   -- clé composite
    FK : id_commande → COMMANDE(id_commande)
    FK : id_produit  → PRODUIT(id_produit)

Analysons ce MLD ligne par ligne :

  • CLIENT et PRODUIT sont des entités simples : une table, une clé primaire, rien de plus (Règle 1).
  • COMMANDE porte #id_client : la clé étrangère qui a migré du côté (1,1) (Règle 2). Le # est la convention Merise pour « clé étrangère ».
  • LIGNE_COMMANDE est née de l'association CONTENIR (Règle 3). Elle réunit les deux clés étrangères plus l'attribut quantite qui appartenait à l'association.
  • Sa clé primaire est composite — formée de deux colonnes (id_commande, id_produit) — car c'est le couple qui est unique : un même produit peut apparaître dans plusieurs commandes, et une commande contient plusieurs produits, mais un produit donné n'apparaît qu'une fois dans une commande donnée.

Correspondance MCD ↔ MLD à mémoriser :

⚠️ Ne confondez pas les niveaux

Dans le MCD on parle d'entités et d'associations ; dans le MLD on parle de tables, de clés primaires et de clés étrangères. Employer « table » dans un MCD ou « association » dans un MLD est l'erreur de vocabulaire la plus fréquente en évaluation.

🎯 Réflexes à garder

🎯 Les 6 pièges à ne pas oublier
  1. Une clé primaire est unique et jamais vide ; deux lignes ne la partagent jamais.
  2. Une clé étrangère pointe vers la PK d'une autre table : c'est le lien qui rend la base « relationnelle ».
  3. Un attribut d'association (comme quantite) ne peut aller dans aucune des deux entités.
  4. Une cardinalité se lit du point de vue de l'entité : min = « obligatoire ? », max = « plusieurs ? ».
  5. La clé migre vers le côté (1,1) ; une association (n,n) devient une table associative.
  6. Bon vocabulaire par niveau : entités/associations au MCD, tables/PK/FK au MLD.
